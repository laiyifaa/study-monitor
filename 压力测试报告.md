# 学习进度监督系统 — 并发压力测试报告

> **测试日期**：2026-07-02（本地）/ 2026-07-03（生产环境）  
> **测试对象**：study-monitor v4.0 — 暑期在线学习平台  
> **测试环境**：Windows 11 本地开发 / Docker 生产环境 (115.223.38.90:1001)

---

## 一、测试背景与目标

### 1.1 系统简介

"学习进度监督系统"是一个基于钉钉 H5 微应用的在线学习平台，核心业务场景为：

- 教师上传课程视频，学生在钉钉内观看学习
- 系统通过心跳机制实时追踪学习时长，并通过多维验证防止刷课
- 支持作业发布、AI 批改、公告推送、排行榜等功能

### 1.2 测试目标

本次压力测试旨在回答一个核心问题：

> **如果 800~1000 名学生同时在线学习，系统后端 API 是否会出现崩溃？**

**明确测试范围**：

| 维度 | 是否实测 | 说明 |
|------|---------|------|
| 心跳 API 吞吐 | **实测** | 每 30 秒/人的心跳上报能否承载 |
| 开始学习 API 并发 | **实测** | 瞬时大量用户同时开始学习时的表现 |
| 登录 API 容量 | **实测** | PBKDF2 密码哈希对 CPU 的冲击 |
| 视频流分发 | **未实测，仅架构分析** | 视频由 Nginx sendfile 直接分发，不经过 Python，详见第六章 |

> **重要说明**：本次测试仅覆盖后端 API 层面（登录、开始学习、心跳上报）。视频流分发依赖 Nginx 内核级 sendfile 机制，该机制不经过 Python 程序，无法在本地开发环境中模拟。视频流相关结论均标注为"架构分析"而非"实测数据"。

## 二、系统架构分析

### 2.1 请求链路

```
钉钉客户端 (学生)
       │
       ▼
   Nginx (:80) ────────────────────────┐
       │                               │
       │ /api/* 代理到 FastAPI          │ /uploads/videos/* 直接分发
       ▼                               ▼
   FastAPI (:8000)              sendfile() 内核零拷贝
       │                          磁盘 → 网卡直达
       ├── Redis (会话限流)         不经过 Python
       └── MySQL (状态持久化)
```

### 2.2 关键配置

| 组件 | 配置项 | 值 |
|------|--------|-----|
| 数据库连接池 | `pool_size` | 20 |
| 数据库连接池 | `max_overflow` | 10 |
| 数据库连接池 | **最大并发连接** | **30** |
| 心跳间隔 | `HEARTBEAT_INTERVAL` | 30 秒 |
| 心跳超时 | `HEARTBEAT_TIMEOUT` | 90 秒 |
| 暂停容忍 | `PAUSE_TOLERANCE` | 300 秒 |
| JWT 有效期 | `JWT_EXPIRE_HOURS` | 72 小时 |
| 密码哈希 | PBKDF2-SHA256 | 100,000 次迭代 |

### 2.3 单次心跳请求的数据库操作

```
1. Redis INCR + EXPIRE   (限流检查，原子操作)
2. SELECT StudySession   (查找活跃会话，主键查询)
3. INSERT HeartbeatLog   (写入心跳日志)
4. UPDATE StudySession   (更新有效学习时长)
5. COMMIT                (事务提交)
```

每次心跳涉及 **4 次数据库操作 + 1 次 Redis 操作**，是系统中频率最高、负载最大的请求。

## 三、测试思路与方法

### 3.1 为什么这样测

本次测试遵循"由浅入深、逐步施压"的原则，设计了三个阶段的测试：

| 阶段 | 测试目的 | 方法 |
|------|---------|------|
| 第一阶段 | 验证测试框架，建立基线 | 50→100→200 人，完整流程（登录→开始→心跳） |
| 第二阶段 | 定位登录瓶颈 | 500 人并发登录，观察 PBKDF2 对 CPU 的影响 |
| 第三阶段 | 核心心跳压测 | 绕过登录瓶颈，直接生成 JWT 进行 1000 人心跳压测 |

**关键设计决策：绕过登录瓶颈**

在第二阶段发现 PBKDF2 密码哈希是 CPU 密集型操作，500 人并发登录即导致系统崩溃。但真实场景中：

- 学生登录是**低频事件**（JWT 有效期 72 小时，每 3 天才登录一次）
- 学生不会**同时登录**（分散在全天不同时段）
- 心跳才是**持续高频负载**（每 30 秒/人，持续整个学习过程）

因此第三阶段通过直接生成 JWT 令牌绕过登录，聚焦于真正决定系统容量的心跳 API。

### 3.2 测试工具

| 工具 | 用途 |
|------|------|
| `concurrency_test.py` | 完整流程测试（登录→开始→心跳→结束） |
| `concurrency_test_1k.py` | 大规模心跳测试（绕过登录，直接生成 JWT） |
| `aiohttp` | 异步 HTTP 客户端，支持高并发请求 |
| `asyncio.Barrier` | 同步屏障，确保所有虚拟用户同时发起请求 |
| `asyncio.Semaphore` | 信号量，控制分批登录的并发数 |

### 3.3 测试指标

| 指标 | 含义 | 判定标准 |
|------|------|---------|
| QPS | 每秒请求数 | 反映系统吞吐能力 |
| P50 延迟 | 50% 请求的响应时间 | 反映典型用户体验 |
| P90 延迟 | 90% 请求的响应时间 | 反映长尾延迟 |
| P99 延迟 | 99% 请求的响应时间 | 反映极端情况 |
| 成功率 | 成功请求 / 总请求 | 核心健康指标 |

| 测试客户端 | 与服务端**同一台机器** | 学生手机远程访问 | 测试脚本抢占 CPU/内存/网络，**测试结果比生产环境更差** |

## 五、测试结果

### 5.1 第一阶段：完整流程基线测试

测试方法：虚拟用户执行完整学习流程（登录 → 开始学习 → 心跳上报），所有用户通过 `asyncio.Barrier` 同步，**同时**发起请求。

| 用户数 | 时长 | 登录 P50 | 开始学习 P50 | 心跳 P50 | 心跳 P99 | 总体成功率 |
|--------|------|----------|-------------|---------|---------|-----------|
| 50 | 30s | 3143ms | 119ms | 4164ms | 4218ms | **100%** |
| 100 | 60s | 3652ms | 829ms | 2086ms | 2114ms | **100%** |
| 200 | 60s | 4920ms | 1842ms | 2078ms | 2328ms | **100%** |

**发现**：

- 200 人以内，系统表现稳定，100% 成功率
- 随着用户数增加，**登录延迟线性增长**（3143ms → 3652ms → 4920ms），这是 PBKDF2 的 CPU 瓶颈
- **开始学习延迟快速增长**（119ms → 829ms → 1842ms），数据库连接池开始出现排队
- **心跳延迟相对稳定**（~2000ms），但整体偏高，说明数据库连接池已接近饱和

### 5.2 第二阶段：登录瓶颈验证

测试方法：500 人同时登录，观察系统表现。

| 用户数 | 登录成功 | 登录失败 | 成功率 |
|--------|---------|---------|--------|
| 500 | 33 | 467 | **6.6%** |

**根因分析**：

PBKDF2-SHA256 每次密码验证需要 100,000 次 SHA-256 迭代，是纯 CPU 计算。Python 的 GIL（全局解释器锁）导致多线程下 CPU 密集型操作被串行化。500 个并发登录请求在事件循环中排队等待 CPU 时间，大量请求超时。

**影响评估**：

在真实场景中，此瓶颈的影响**有限**，因为：
- JWT 有效期 72 小时，学生每 3 天才登录一次
- 登录时间天然分散（全天不同时段）
- 生产环境（Linux + Docker）可通过部署多个 Uvicorn worker 进程分担 CPU 负载

### 5.3 第三阶段：核心心跳压力测试

测试方法：绕过登录，直接通过 PyJWT 库生成 JWT 令牌，聚焦于心跳 API 的并发承载能力。同时引入**分散开始（stagger）**策略，模拟真实场景中学生陆续开始学习的行为。

#### 5.3.1 同时开始（stagger=0s）

| 用户数 | 开始学习 | 心跳 | 总体成功率 |
|--------|---------|------|-----------|
| 200 | 200/200 (100%) | 200/200 (100%) | **100%** |
| 1000 | 487/1000 (48.7%) | 917/974 (94.1%) | **71.1%** |

**分析**：1000 人同时开始学习时，1000 个数据库请求瞬间涌入，争抢 **30 个数据库连接**。连接池已被占满，后续请求排队等待，最终超时。

#### 5.3.2 分散 5 秒开始（stagger=5s）

| 用户数 | 开始学习 | 心跳 | 总体成功率 |
|--------|---------|------|-----------|
| 1000 | 870/1000 (87.0%) | 1655/1740 (95.1%) | **92.2%** |

**分析**：分散 5 秒相当于将瞬时 1000 请求削峰为每秒约 200 请求。成功率从 48.7% 提升至 87.0%，但仍有部分超时，说明连接池仍有压力。

#### 5.3.3 分散 10 秒开始（stagger=10s）— 真实场景模拟

| 用户数 | 开始学习 | 心跳 | 总体成功率 |
|--------|---------|------|-----------|
| 1000 | **1000/1000 (100%)** | **2000/2000 (100%)** | **100%** |

**详细指标**：

| 指标 | 开始学习 API | 心跳上报 API |
|------|-------------|-------------|
| 总请求数 | 1000 | 2000 |
| 成功率 | **100%** | **100%** |
| QPS | 16.7 | 33.3 |
| P50 延迟 | 2458ms | 2032ms |
| P90 延迟 | 2636ms | 2093ms |
| P99 延迟 | 2686ms | 2137ms |
| 平均延迟 | 2442ms | 1251ms |
| 最小延迟 | 2066ms | 17ms |
| 最大延迟 | 2902ms | 2159ms |

**3000 次 API 请求，零失败。**

**需要关注的延迟**：心跳 P50 延迟为 2032ms，意味着半数请求等待 2 秒以上。但这主要受测试环境限制（测试脚本与服务端抢占同一台机器的 CPU 和数据库连接），生产环境预期延迟会显著降低。此外，心跳是后台静默发送的，学生端感知不到这 2 秒的延迟。

## 六、瓶颈分析

### 6.1 瓶颈一：数据库连接池（主要瓶颈，实测确认）

```
┌─────────────────────────────────────────────────────┐
│                    数据库连接池                       │
│  ┌───┐ ┌───┐ ┌───┐ ... ┌───┐  ← 20 个常驻连接       │
│  └───┘ └───┘ └───┘     └───┘                        │
│  ┌───┐ ... ┌───┐  ← 10 个溢出连接                   │
│  └───┘     └───┘                                    │
│                                                      │
│  总计：30 连接                                       │
│  实测：1000 人同时开始 → 48.7% 成功率（连接池不足）     │
│  实测：1000 人分散 10s 开始 → 100% 成功率（可行）      │
└─────────────────────────────────────────────────────┘
```

**连接池需求推算**（基于实测数据）：

- 1000 人同时开始，平均延迟 3961ms，30 连接理论最大并发处理能力 ≈ 30 × (1000ms / 平均每请求处理时间)，实际远低于 1000 并发需求
- 1000 人分散 10s，每秒约 100 请求，平均每个请求占用连接约 2.5s，理论需求 ≈ 100 × 2.5 ≈ 250 连接·秒，30 连接可在约 8 秒内处理完，落在超时阈值内

**影响**：同时开始的瞬时并发会被连接池限制，导致请求排队超时。  
**缓解**：学生自然分散开始时间（10 秒即可完全消除问题）。  
**源码级优化建议**（需修改 `backend/app/database.py`）：将 `pool_size` 从 20 提升至 40，`max_overflow` 从 10 提升至 20，总连接数 60，可应对 2000 人同时场景。

### 6.2 瓶颈二：PBKDF2 密码哈希（次要瓶颈，实测确认）

**影响**：500 人同时登录成功率仅 6.6%。  
**真实风险**：低。登录是低频事件，JWT 72 小时有效。  
**缓解措施**（无需修改源码，仅调整部署方式）：
- 生产环境部署时使用 `gunicorn` 多 worker 进程，分散 CPU 负载
- 此瓶颈在本次测试中被绕过（第三阶段直接生成 JWT），不影响核心心跳测试结论

### 6.3 视频流分发（架构分析，非实测）

> **本节内容基于 nginx 配置文件分析，未经实际测试验证。**

**架构保障**：生产环境中，视频由 Nginx 通过 `sendfile()` 系统调用直接分发。数据从磁盘到网卡在内核态完成，**不经过 Python 解释器，不经过 FastAPI 应用层**。

**五层视频分发保障**（基于 `nginx/default.conf` 分析）：

| 层级 | 技术 | 作用 |
|------|------|------|
| 1 | Nginx sendfile | 内核零拷贝，数据从磁盘到网卡不经过用户态 |
| 2 | MP4 流模块 | 支持 Range 请求，拖动进度条只传输需要的片段 |
| 3 | 浏览器缓存 | `expires 7d` + `immutable`，已观看视频不再重复下载 |
| 4 | 文件句柄缓存 | `open_file_cache max=200`，热门视频元数据驻留内存 |
| 5 | CDN 加速（可选） | 配置 `CDN_DOMAIN` 后，视频通过天翼云 CDN 就近分发 |

**理论带宽估算**（基于项目作者在 nginx 配置中的注释）：

```
单路 1080p 视频流 ≈ 4.5 Mbps
800 人并发 ≈ 3.6 Gbps
Nginx sendfile 在 4 核 8G NVMe SSD 服务器上可达线速
瓶颈在机房带宽，不在服务器软件
```

**待验证项**：上述估算未在本次测试中验证。建议上线后在服务器上使用 `wrk` 或 `ab` 对 Nginx 视频 URL 做独立的静态文件压力测试。

## 七、结论与建议

### 7.1 核心结论

| 场景 | 结论 | 依据 |
|------|------|------|
| 200 人同时在线 | **完全没问题**，100% 成功率 | 实测 |
| 1000 人同一瞬间点播放 | 会崩溃（48.7% 成功率），连接池不足 | 实测 |
| 1000 人 10 秒内陆续开始 | **完全没问题**，100% 成功率，3000 次请求零失败 | 实测 |
| 1000 人同时心跳上报 | 能承载（33.3 QPS），延迟偏高（P50: 2s）但无失败 | 实测 |

### 7.2 真实场景评估

暑期在线学习平台，800 名学生同时上课，他们的行为模式是：

```
上课铃响 → 学生陆续打开钉钉 → 进入课程 → 点击播放视频
         └── 这个过程自然分散在 30 秒 ~ 2 分钟内 ──┘
```

在这种自然分散模式下：

- **API 层面**：实测数据表明 1000 人分散 10 秒即可 100% 成功，800 人分散 30 秒~2 分钟只会更稳定
- **视频流层面**：未实测，但 Nginx sendfile 是业界成熟方案，理论带宽需求（800 人 × 4.5Mbps ≈ 3.6Gbps）在典型服务器配置下可满足

### 7.3 优化建议

| 优先级 | 建议 | 类型 | 预期效果 |
|--------|------|------|---------|
| **高** | 生产环境使用 `gunicorn` 多 worker 部署 | 部署调整 | 提升 CPU 密集型任务吞吐，缓解登录瓶颈 |
| **高** | 上线后在服务器上对 Nginx 视频 URL 做独立压测 | 补充测试 | 验证视频流分发是否能承载 800 人 |
| **中** | 数据库连接池扩容至 60（pool_size=40, max_overflow=20） | **需改源码** | 可承载 2000 人瞬时并发 |
| **中** | 添加 API 限流中间件（如 `slowapi`） | **需改源码** | 防止恶意请求击穿系统 |
| **低** | 为心跳 API 添加 Redis 缓存层 | **需改源码** | 降低心跳延迟 50%+ |
| **低** | 配置 CDN_DOMAIN 启用天翼云 CDN | 配置调整 | 视频就近分发，降低源站带宽压力 |

### 7.4 测试脚本

本次测试使用的脚本位于项目根目录：

| 脚本 | 用途 | 环境 |
|------|------|------|
| `concurrency_test.py` | 完整流程压力测试（含登录） | 本地 |
| `concurrency_test_1k.py` | 大规模心跳压力测试（绕过登录） | 本地 |
| `stress_test_prod.py` | 800 人完整流程测试（API 创建用户 + 登录 + 学习 + 心跳） | 生产环境 |

使用方式：


### 8.1 测试说明

本地测试完成后，项目已部署至公网服务器（Docker 生产环境），进行了第二轮独立验证。

| 项目 | 详情 |
|------|------|
| 服务器地址 | 115.223.38.90:1001 |
| 部署方式 | Docker Compose（backend + frontend + MySQL + Redis） |
| 测试日期 | 2026-07-03 |
| 测试用户数 | 800 |
| 测试工具 | `stress_test_prod.py`（Python asyncio + aiohttp） |
| 测试策略 | 通过管理员 API 创建 800 个测试学生，执行完整学习流程 |

### 8.2 测试流程

```
阶段 1: 创建 800 个测试学生（通过管理员 API 批量创建）
阶段 2: 800 人同时登录（asyncio.Barrier 同步）
阶段 3: 800 人开始学习 + 60 秒心跳循环（分散 10 秒开始）
阶段 4: 清理测试数据
```

### 8.3 测试结果

| 阶段 | 请求数 | 成功 | 成功率 | 耗时 |
|------|--------|------|--------|------|
| 创建用户 | 800 | 800 | **100%** | 55.6s |
| 同时登录 | 800 | 800 | **100%** | 52.9s |
| 开始学习 | 800 | 800 | **100%** | — |
| 心跳上报 | 800 | 800 | **100%** | — |

#### 登录延迟

| 指标 | 值 |
|------|-----|
| P50 | 26,797ms |
| P90 | 47,886ms |
| P99 | 52,803ms |

**分析**：800 人同时登录时，PBKDF2 密码哈希（10 万次迭代）在 Docker 单容器中成为 CPU 瓶颈。但登录是低频事件（JWT 72 小时有效），且真实场景中 800 人不会在同一秒登录。

| 平均 | 50ms |

| P90 | 81ms |

### 8.4 与本地测试的对比

| 指标 | 本地测试 (1000 人) | 生产环境 (800 人) | 差异分析 |
|------|-------------------|-------------------|---------|
| 开始学习 P50 | 2458ms | **43ms** | 生产环境快 57 倍 |
| 开始学习 P99 | 2686ms | **132ms** | 生产环境快 20 倍 |
| 心跳 P50 | 2032ms | **52ms** | 生产环境快 39 倍 |
| 心跳 P99 | 2137ms | **145ms** | 生产环境快 15 倍 |
| 心跳 QPS | 33.3 | 11.4 | 用户数差异（1000 vs 800） |
| 成功率 | 100% | **100%** | 一致 |

**关键发现**：生产环境的 API 延迟比本地测试低一个数量级。原因：

- **本地测试**：测试脚本与服务器共享同一台机器的 CPU、内存、磁盘 IO，测试脚本本身占用了大量系统资源，导致 API 延迟被人为抬高
- **生产环境**：测试脚本运行在远程客户端，服务器资源完全用于处理 API 请求，延迟反映真实用户体验

这验证了第四章提出的判断——**本地测试结果偏保守，生产环境表现明显更好**。

### 8.5 生产环境结论

| 场景 | 结论 | 依据 |
|------|------|------|
| 800 人同时在线学习 | **完全没问题** | P99 延迟 < 150ms，100% 成功率 |
| 800 人同时登录 | CPU 瓶颈（P50=27s），但实际场景不会同时发生 | 实测 |
| 视频播放流畅度 | 依赖 Nginx sendfile 架构，未在 API 测试中覆盖 | 架构分析 |
